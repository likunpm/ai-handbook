# 3. 产品 Agent 化：软件形态正在发生什么变化

> 软件的交互范式正在经历一次根本性的迁移，从图形界面到语言界面，从工具到代理。

---

## GUI 时代：LUI 分发

过去十年，移动互联网的核心逻辑是 **GUI（图形用户界面）**。

用户打开 App，看到的是一个精心设计的界面：导航栏、按钮、卡片、列表。每一个交互都是设计师预设好的路径，用户在这些路径里做选择。

这套逻辑里，产品的核心工作是：
- 设计清晰的信息架构
- 优化用户的点击路径
- 降低认知负担

**LUI（语言用户界面）的出现，开始瓦解这套逻辑。**

用户不再需要找到正确的按钮，他们直接说出需求："帮我订一张明天去上海的票"、"给我推荐一个适合初学者的训练计划"。

界面不再是产品的主体，**语言成了新的入口。**

这对 PM 的挑战是：你过去积累的界面设计经验，在 LUI 世界里价值有限。新的核心能力是：如何设计一个能准确理解用户意图的对话系统。

---

## GUI 时代：Copilot

LUI 没有直接替代 GUI，而是先以 **Copilot** 的形态嵌入进了现有产品。

Copilot 的特点是：**人在主导，AI 在辅助。**

典型形态：
- Word 里的 AI 写作助手
- IDE 里的代码补全（GitHub Copilot）
- Excel 里的公式生成
- 设计工具里的图层自动命名

这个阶段，产品的 GUI 结构基本不变，AI 作为一个"侧边栏"或"浮层"嵌入。用户还是在用熟悉的界面完成任务，AI 降低了某些操作的成本。

**Copilot 模式的局限**在于：它只能优化已有流程，不能重构流程本身。用户还是需要知道要做什么，AI 只是帮你做得更快。

---

## AI Native 时代：Skill 时代

真正的范式转变发生在 **Agent** 身上。

Agent 不只是回答问题，它能**完成任务**。它有记忆、能调用工具、能执行多步骤操作、能在完成任务的过程中做判断。

在 Agent 时代，软件的形态发生了根本变化：

**从"功能"到"Skill"**

过去的软件逻辑：用户打开 App → 找到功能 → 执行操作。

Agent 时代的逻辑：用户描述目标 → Agent 调用所需的 Skill → 完成任务。

Skill 是可被 Agent 调用的能力单元。搜索是一个 Skill，预订是一个 Skill，生成图片是一个 Skill，发送邮件是一个 Skill。Agent 把这些 Skill 串起来，完成用户的复合目标。

**这意味着什么？**

- 产品的竞争单位从"App"变成了"Skill 的质量和覆盖度"
- 谁的 Skill 更好、更多、更可靠，谁就更容易成为 Agent 调用的首选
- 流量的入口从"应用图标"转移到了"Agent 的意图路由"

**对 PM 的影响**

以前做产品，你设计的是用户的完整使用路径。
现在做产品，你需要思考：**你的产品如何成为一个好的 Skill，如何被 Agent 正确地调用、正确地返回结果。**

这是一个全新的产品设计维度，大多数 PM 还没有真正面对它。

---

## 三个阶段的对比

| 阶段 | 交互方式 | 人与 AI 的关系 | PM 的核心工作 |
|---|---|---|---|
| GUI + LUI 分发 | 图形界面 + 搜索/推荐 | AI 负责分发，人负责消费 | 界面设计、信息架构 |
| Copilot | 图形界面内嵌 AI 助手 | 人主导，AI 辅助执行 | 场景植入、效果评估 |
| AI Native / Agent | 自然语言 + Skill 调用 | AI 主导任务执行，人负责目标定义和验收 | 任务边界设计、Skill 开发、意图识别 |

我们正处在从 Copilot 向 Agent 过渡的阶段。

不是所有产品都会变成 Agent，但几乎所有产品都会被 Agent 的逻辑影响。
